00040 /**
00041  * This class persists a log on the filesystem efficiently.
00042  *
00043  * The log entries on disk are stored in a series of files called segments, and
00044  * each segment is about 8MB in size. Thus, most small appends do not need to
00045  * update filesystem metadata and can proceed with a single consecutive disk
00046  * write.
00047  *
00048  * The disk files consist of metadata files, closed segments, and open
00049  * segments. Metadata files are used to track Raft metadata, such as the
00050  * server's current term, and also the log's start index. Segments contain
00051  * contiguous entries that are part of the log. Closed segments are never
00052  * written to again (but may be renamed and truncated if a suffix of the log is
00053  * truncated). Open segments are where newly appended entries go. Once an open
00054  * segment reaches MAX_SEGMENT_SIZE, it is closed and a new one is used.
00055  *
00056  * Metadata files are named "metadata1" and "metadata2". The code alternates
00057  * between these so that there is always at least one readable metadata file.
00058  * On boot, the readable metadata file with the higher version number is used.
00059  *
00060  * Closed segments are named by the format string "%020lu-%020lu" with their
00061  * start and end indexes, both inclusive. Closed segments always contain at
00062  * least one entry; the end index is always at least as large as the start
00063  * index. Closed segment files may occasionally include data past their
00064  * filename's end index (these are ignored but a WARNING is issued). This can
00065  * happen if the suffix of the segment is truncated and a crash occurs at an
00066  * inopportune time (the segment file is first renamed, then truncated, and a
00067  * crash occurs in between).
00068  *
00069  * Open segments are named by the format string "open-%lu" with a unique
00070  * number. These should not exist when the server shuts down cleanly, but they
00071  * exist while the server is running and may be left around during a crash.
00072  * Open segments either contain entries which come after the last closed
00073  * segment or are full of zeros. When the server crashes while appending to an
00074  * open segment, the end of that file may be corrupt. We can't distinguish
00075  * between a corrupt file and a partially written entry. The code assumes it's
00076  * a partially written entry, issues a WARNING, and ignores it.
00077  *
00078  * Truncating a suffix of the log will remove all entries that are no longer
00079  * part of the log. Truncating a prefix of the log will only remove complete
00080  * segments that are before the new log start index. For example, if a
00081  * segment has entries 10 through 20 and the prefix of the log is truncated to
00082  * start at entry 15, that entire segment will be retained.
00083  *
00084  * Each segment file starts with a segment header, which currently contains
00085  * just a one-byte version number for the format of that segment. The current
00086  * format (version 1) is just a concatenation of serialized entry records.
00087  */
